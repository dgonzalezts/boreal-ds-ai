# Generator Extension Strategy: Cross-Package Component Scaffolding

## Executive Summary

This document outlines the strategy to extend the Plop.js generator from **Storybook stories only** (in `apps/boreal-docs`) to support **both Storybook stories AND Stencil web components** (in `packages/boreal-web-components`).

**Current State**: Plop generates `.stories.ts` + `.mdx` files in `apps/boreal-docs`
**Future State**: Single generator that can scaffold:

1. Stencil component files in `packages/boreal-web-components`
2. Corresponding Storybook stories in `apps/boreal-docs`
3. Both simultaneously (recommended workflow)

---

## Package Structure Comparison

### Storybook Stories (apps/boreal-docs)

**Technology**: Lit templates, Storybook 10.1.11
**Files per component**:

```
src/stories/{Category}/{ComponentName}/
├── {ComponentName}.stories.ts     # Story configuration + variants
└── {ComponentName}.mdx            # Documentation
```

**Purpose**: Documentation and interactive demos

---

### Web Components (packages/boreal-web-components)

**Technology**: Stencil.js (Web Components compiler)
**Files per component**:

```
src/components/{component-name}/
├── {component-name}.tsx           # Stencil component implementation
├── {component-name}.css           # Component styles
├── {component-name}.spec.ts       # Unit tests (Stencil testing)
├── {component-name}.e2e.ts        # E2E tests (Puppeteer)
└── readme.md                      # Auto-generated by Stencil build
```

**Purpose**: Production web component library

**Build Outputs**:

- React wrapper → `packages/boreal-react/lib/components/` (auto-generated)
- Vue wrapper → `packages/boreal-vue/lib/components.ts` (auto-generated)
- TypeScript definitions, loaders, custom elements bundles

## Extension Options Analysis

### Option 1: Separate Generators (Recommended for Phase 1)

**Architecture**: Two independent generators in their respective packages

```
apps/boreal-docs/
├── plopfile.js              # Story generator only
└── .plop-templates/
    └── story-simple/

packages/boreal-web-components/
├── plopfile.js              # Component generator only
└── .plop-templates/
    └── component/
```

**Workflow**:

```bash
# Generate web component
cd packages/boreal-web-components
pnpm generate:component

# Generate story for that component
cd ../../apps/boreal-docs
pnpm generate:story
```

**Pros**:

- ✅ Simple to implement (independent plopfiles)
- ✅ Clear separation of concerns
- ✅ No cross-package coordination needed
- ✅ Can implement incrementally
- ✅ Easier to test and maintain

**Cons**:

- ❌ Two-step process for full component + story
- ❌ Potential naming inconsistencies
- ❌ Duplicate prompts (component name twice)

**Recommendation**: ⭐ **Start here** - Implement both generators independently, then optionally unify in Option 2

---

### Option 2: Unified Generator (Future Enhancement)

**Architecture**: Single root-level generator that creates files in both packages

```
tools/
└── generators/
    ├── plopfile.js          # Unified configuration
    └── .plop-templates/
        ├── component/       # Stencil component templates
        └── story/           # Storybook story templates
```

**Workflow**:

```bash
# From project root
pnpm generate

# Single prompt flow:
? Component name: MyButton
? Generate Stencil component? (Y/n) Y
? Generate Storybook story? (Y/n) Y
? Component location: packages/boreal-web-components/src/components
? Story category: Components
```

**Pros**:

- ✅ Single command for both outputs
- ✅ Guaranteed naming consistency
- ✅ Shared prompts (DRY principle)
- ✅ Better developer experience

**Cons**:

- ❌ More complex plopfile logic
- ❌ Cross-package path management
- ❌ Harder to maintain if packages diverge
- ❌ Requires root-level tooling setup

**Recommendation**: ⏭️ **Phase 2** - After both independent generators are proven and stable

---

### Option 3: Orchestrator Script (Most Advanced)

**Architecture**: Custom Node.js script that invokes both generators sequentially

```
tools/
└── scripts/
    └── generate-full-component.js    # Orchestrates both generators
```

**Workflow**:

```bash
pnpm generate:full

# Script internally runs:
# 1. cd packages/boreal-web-components && plop component
# 2. cd apps/boreal-docs && plop story
# Both with shared data
```

**Pros**:

- ✅ Reuses existing generators (no duplication)
- ✅ Single command experience
- ✅ Can add extra logic (validation, post-generation tasks)

**Cons**:

- ❌ Custom script maintenance
- ❌ More moving parts
- ❌ Harder to debug
- ❌ Overkill for current needs

**Recommendation**: ❌ **Not recommended** - Too complex for minimal benefit

## Technical Considerations

### 1. Naming Conventions

**Challenge**: Different naming between packages

- Stencil components: `my-button` (kebab-case tag, PascalCase class)
- Storybook stories: `MyButton` (PascalCase files)

**Solution**: Use PascalCase input, transform to kebab-case where needed

```javascript
// In plopfile prompts
name: 'componentName'  // User enters: "MyButton"

// In templates
{{pascalCase componentName}}   → MyButton
{{kebabCase componentName}}    → my-button
{{camelCase componentName}}    → myButton
```

### 2. Path Resolution

**Challenge**: Generators in different packages need different base paths

**Solution (Option 1 - Separate Generators)**:

- Each plopfile has relative paths from its location
- `apps/boreal-docs/plopfile.js` → paths relative to `apps/boreal-docs/`
- `packages/boreal-web-components/plopfile.js` → paths relative to `packages/boreal-web-components/`

**Solution (Option 2 - Unified Generator)**:

```javascript
// In root tools/generators/plopfile.js
const PROJECT_ROOT = process.cwd();
const DOCS_PATH = path.join(PROJECT_ROOT, "apps/boreal-docs");
const COMPONENTS_PATH = path.join(
  PROJECT_ROOT,
  "packages/boreal-web-components",
);

// Use absolute paths in actions
path: path.join(
  COMPONENTS_PATH,
  "src/components/{{kebabCase componentName}}/...",
);
```

### 3. Build System Integration

**Challenge**: Stencil auto-generates files on build (readme.md, React/Vue wrappers)

**Solution**:

- Include placeholder readme.md in generator
- Document that React/Vue wrappers generate automatically
- Add note in post-generation message: "Run `pnpm build` to generate framework wrappers"

### 4. Cross-Package Consistency

**Challenge**: Keeping component name consistent between packages

**Solution (Option 1)**:

- Document naming convention in both generators
- Manual user responsibility to use same name

**Solution (Option 2)**:

- Unified generator ensures consistency automatically
- Single prompt feeds both template sets

### 5. Dependencies

**Challenge**: Each package needs its own plop installation (no monorepo orchestrator)

**Solution (Option 1)**:

```bash
# Install in each package separately
cd apps/boreal-docs
pnpm add -D plop

cd ../../packages/boreal-web-components
pnpm add -D plop
```

**Solution (Option 2 - Root-level)**:

```bash
# Install at root (if moving to unified)
cd /Users/dgonzalez/projects/src/boreal-ds
pnpm add -D plop

# Add "tools" workspace to package.json
{
  "workspaces": [
    "apps/*",
    "packages/*",
    "tools"
  ]
}
```

### Migration Step 3: Create Unified Plopfile

Create `tools/generators/plopfile.js`:

```javascript
import path from "path";

export default function (plop) {
  const PROJECT_ROOT = process.cwd();

  // Register helpers
  plop.setHelper("titlePath", (category, name) => `${category}/${name}`);

  // Main generator with choice
  plop.setGenerator("component", {
    description: "Generate web component and/or Storybook story",
    prompts: [
      {
        type: "list",
        name: "generateWhat",
        message: "What do you want to generate?",
        choices: [
          { name: "Both (Component + Story)", value: "both" },
          { name: "Web Component only", value: "component" },
          { name: "Storybook Story only", value: "story" },
        ],
        default: "both",
      },
      {
        type: "input",
        name: "componentName",
        message: "Component name (PascalCase):",
        validate: (value) => (value ? true : "Required"),
      },
      {
        type: "input",
        name: "description",
        message: "Description:",
        default: (answers) =>
          `A ${answers.componentName} component for the Boreal Design System`,
      },
      {
        type: "list",
        name: "category",
        message: "Story category:",
        when: (answers) => answers.generateWhat !== "component",
        choices: ["Components", "Forms", "Layout", "Navigation", "Other"],
        default: "Components",
      },
    ],
    actions: (data) => {
      const actions = [];
      const shouldGenerateComponent = ["both", "component"].includes(
        data.generateWhat,
      );
      const shouldGenerateStory = ["both", "story"].includes(data.generateWhat);

      // Component actions
      if (shouldGenerateComponent) {
        const componentBase = path.join(
          PROJECT_ROOT,
          "packages/boreal-web-components/src/components",
        );
        actions.push(
          {
            type: "add",
            path: path.join(
              componentBase,
              "{{kebabCase componentName}}/{{kebabCase componentName}}.tsx",
            ),
            templateFile: ".plop-templates/component/component.tsx.hbs",
          },
          // ... (other component files)
        );
      }

      // Story actions
      if (shouldGenerateStory) {
        const storyBase = path.join(
          PROJECT_ROOT,
          "apps/boreal-docs/src/stories",
        );
        actions.push(
          {
            type: "add",
            path: path.join(
              storyBase,
              "{{category}}/{{componentName}}/{{componentName}}.stories.ts",
            ),
            templateFile:
              ".plop-templates/story-simple/Component.stories.ts.hbs",
          },
          // ... (other story files)
        );
      }

      return actions;
    },
  });
}
```

## Risks & Mitigations

### Risk 1: Template Drift

**Risk**: Templates diverge from actual patterns over time

**Mitigation**:

- Document update process in README
- Quarterly review of templates vs actual components
- Version control tracks changes
- Reference components in template comments

### Risk 2: Cross-Package Naming Inconsistencies

**Risk**: User creates `MyButton` component but `MyBtn` story

**Mitigation (Phase 1)**:

- Document naming conventions clearly
- Code review process catches mismatches

**Mitigation (Phase 2)**:

- Unified generator enforces consistency automatically

### Risk 3: Build System Changes

**Risk**: Stencil or Storybook updates break templates

**Mitigation**:

- Pin plop version
- Test generators after major upgrades
- Keep templates simple (less surface area for breaks)
- Document template update process

### Risk 4: Maintenance Burden

**Risk**: Multiple plopfiles to maintain

**Mitigation (Phase 1)**:

- Accept trade-off for simpler implementation
- Both plopfiles are similar (easy to sync changes)

**Mitigation (Phase 2)**:

- Migrate to unified generator when stable
